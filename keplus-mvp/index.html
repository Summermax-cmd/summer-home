Ke+ 房源“广告”投放系统

把房源扶持从「按需按次开发、曝光驱动」升级为「目标驱动、效率约束、自动止损、结果可归因」的内部房源“广告”投放系统。

项目背景与产品定位

过去平台更像一个货架——把房源摆上去、靠自然曝光等待匹配。但进入买方市场后，当我们想快速去化某一类房子（业主去化、诚意卖、主理人、重点运营房源等），传统做法只能在已有货架上「定点定需求」地逐个开发坑位：投放天花板低、时间成本高、每来一类需求都要重新开发一次，成本居高不下。

我们需要一种新能力——在市场动态下，以目标驱动的方式稳定、可复用地完成某类房源的快速去化。为此建设一套面向房源访问的内部“广告”投放系统。

① 目标不可控

需求方希望获得一定点击规模，但系统无法稳定保障周期点击达成。

② 效果不可归因

投放后只看到房源点击量，难以判断是否由扶持带来。

③ 停止机制不清

房源表现差时缺少自动止损，浪费流量并影响用户体验。

④ 场景策略不统一

搜索、推荐、首页、小区页对扶持接受度不同，缺统一框架。

⑤ 运营成本高

每次扶持都靠人工判断、加权、复盘，难以规模化。

过去 · 货架曝光 + 定点开发
  • 平台即货架，房源摆上去靠自然曝光
  • 要扶持只能在已有货架上「定点定需求」开发坑位
  • 投放天花板低、时间成本高、一次次重复开发
  • 只保证曝光，不保证去化目标，无统一止损
→
现在 · 房源“广告”投放系统
  • 以周期核访总量为目标，自动达成
  • 以 pCTR 与场景效率为约束
  • 以自动投放 + 自动止损为核心
  • 报表结构化归因，可规模复用
这不是外部商业“广告”系统，而是一个内部房源流量扶持系统——借鉴计算“广告”的投放思想，服务对象是贝壳内部业务中的目标房源。需求方提供要扶持的房源和周期核访目标，Ke+ 投放系统负责在多个分发场景中自动完成核访目标，并在点击效率过低时自动止损。

MVP 目标与边界

MVP 先跑通一个最小闭环：需求方提供房源 → 系统完成周期核访目标 → 自动止损 → 报表归因。

核心业务流程

从需求方创建任务，到在线分发排序，再到自动停止与归因报表的完整链路。

创建投放任务 → 传入房源 + 周期目标 → 生成投放任务 → 待投放 → 投放中 → 场景请求召回目标房源 → 计算 pCTR · Boost · Pacing → 参与排序 / 注入 → 记录曝光 · 点击 · 核访 → 实时监控完成率 · 效率 → 达成 → 停止 效率过低 → 止损 → 任务报表 + 停止归因

在线投放逻辑与关键因子

MVP 尽量走白盒逻辑：放置规则决定「房源放在哪个坑」，关键因子决定「出哪些、按什么顺序、出多少」，出完本次投放候选即停，规则清晰、可解释、可复盘。

① 白盒放置规则 · 坑位注入 / 置顶

规则一 · 坑位注入

推荐 feeds / 二手首页房源列表

  • 一刷信息流中，所有偶数位（第 2、4、6… 位）均可出扶持房源
  • 按投放候选顺序依次填入偶数坑，直到本次投放候选出完
  • 不改动自然排序，仅占用偶数坑位
  • 二手首页注入限非首屏
·
规则二 · 置顶

搜索结果 / 小区详情页房源列表

  • 命中投放的房源先全部置顶展示
  • 按投放候选顺序依次出，直到本次投放候选出完
  • 之后回归自然结果，不再干预
  • 小区详情页仅置顶当前小区内目标房源
强上下文场景底线（搜索 / 筛选）：搜索结果与筛选页意图最强，扶持有硬性底线——搜索必须与 query 相关；筛选后必须符合用户的筛选条件。不满足则该房源不出，宁可不投也不破坏上下文。

② 关键因子 · 决定出哪些 / 顺序 / 出多少

Relevance 作硬性底线过滤掉不该出的房源，pCTR × Boost 给通过底线的投放候选排序，Pacing 按目标进度控制本次出多少、是否暂缓。

AdScore (eCPM) = pCTR × Boost
Relevance 作 0/1 底线先过滤，未过线不进候选 pCTR × Boost 给候选排序，分高者先出 Pacing 控制本次出多少 / 是否暂缓

pCTR · 是否值得投

  • 优先接入现有推荐 / 搜索 pCTR 模型
  • 无实时 pCTR 时用历史 CTR 近似
  • 小样本做平滑：
    (点击 + α×场景均值) / (曝光 + α)

Boost · 扶持强度

  • Boost = TaskPriority × HouseWeight × Pacing
  • 任务优先级权重
  • 需求方传入房源 weight
  • 节奏带来的加速 / 降速

Relevance · 相关性

  • 搜索：城市一致 + 满足筛选 + 不偏离
  • feeds：不与负偏好冲突、匹配近期偏好
  • 二手首页：首屏更严、非首屏稍宽
  • 小区页：房源必须属于当前小区

Pacing · 节奏控制

  • 目标完成率 = 累计核访 / 周期目标
  • 时间消耗率 = 已投时长 / 总周期
  • 对比两者，调整 PacingBoost

Pacing 判断逻辑

状态动作
目标完成率 < 时间消耗率提高 PacingBoost
目标完成率 ≈ 时间消耗率维持
目标完成率 > 时间消耗率降低 PacingBoost
目标完成率 ≥ 100%停止投放

核访目标与停止机制

目标口径 周期核访总量 仅设在任务级目标（核访量），不强制每套房源都有目标
核访口径 点击进入投放房源详情 用户点击进入被投放房源的详情页即计为一次核访；口径可降级为点击行为
房源维度只做 统计 · cap · 识别 核访贡献统计、单房源 cap、低效房源识别与异常下线
示例目标 7 天 / 100 万 7 天内为 500 套目标房源累计带来 100 万次核访
① 目标达成停止
累计核访量 ≥ 目标核访量
  • 任务状态变更为已达成，停止原因 TARGET_REACHED
  • 记录停止时间与最终核访量
  • 所有目标房源停止扶持投放
  • 目标房源回归自然排序
② 点击效率过低止损
扶持拉低场景整体核访率（即 CTR）
  • 采用场景级止损 + 任务级止损两层机制
  • 所有可投场景被止损 → 任务停止
  • 停止原因 ALL_SCENES_STOPPED_BY_LOW_EFFICIENCY
  • 避免为完成目标而持续消耗低效流量

效率止损规则

止损只看一类触发：扶持是否拉低场景整体核访率（即 CTR）。对比开启扶持后的场景整体核访率与基准，下降超过 10% 即止损；判断须满足最小样本量，避免小样本误判。

核访率下降率 = （基准核访率 − 当前核访率） ÷ 基准核访率
下降率 ≥ 5% → 场景限速 下降率 > 10% → 停止该场景投放 基准取自预留 10% 流量（不投扶持的对照组）

整体核访率下降止损底线

统一底线 · 四场景一致 下降 > 10%

≈ 意味着商机潜在约降 10%

任一可投场景整体核访率较基准下降超过 10%，即停止该场景投放；下降达 5%（底线一半）先场景限速观察。不再按场景设差异化阈值。

最小样本量要求（未达到只观察）

样本太少时核访率波动大、易误判，曝光与核访都达到门槛后才触发止损；未达门槛只统计观察、不止损。
